what it is

A small platform pattern for agent operations.

The repo models the operational layer around tool-using AI agents: exact-release promotion, rollback proof, signed deployment artifacts, runtime isolation, and controls that scale with blast radius.

It is deliberately agent-agnostic. A workload enters as a manifest and a risk tier, then lands in one of two abstract deployment archetypes: immutable-release service or package-install agent pool.

why the boundary exists

Agents are untrusted because their inputs can become behavior.

01

Tool calls under model control

Shell, files, network, tickets, messages, and other agents are side effects, not chat.

02

Credential-bearing runtime

Secrets are references, never page content. Authority is scoped and auditable.

03

Prompt injection surface

External content is treated as data. Tool allowlists fail closed when policy is absent.

04

Shared infrastructure drift

Status re-derives truth. Promotion is exact commit; mutation starts as dry-run.

three layers + governance

Every layer assumes the one above it can fail.

01

Promotion control plane

Immutable exact-commit releases, dry-run-by-default mutation, drift detection, state as truth, and recorded rollback targets.

  • exact commit promotion
  • dry-run before apply
  • drift-aware status
02

Isolation substrate

Nested-virt golden VM with Tier-1 cosign-signed, digest-pinned services and Tier-2 ephemeral Kata microVM jobs.

  • reconcile → align
  • default-deny egress
  • verified teardown
03

agent-gateway@<profile>

Per-profile runtime layout with one declared runtime source, OS-level profile isolation, and independently restartable hardened systemd units.

  • one service per profile
  • Linux user isolation
  • restart blast-radius control
G

Governance overlay

Risk tiers L0–L5, SLO/canary, fail-closed MCP tool allowlists, secrets-by-reference, signed supply chain, audit trails, and proven rollback.

  • risk-tier gates
  • canary before promotion
  • audit evidence
